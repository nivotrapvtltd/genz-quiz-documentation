Welcome

Everything needed to install, deploy, and extend Genz Quiz

Genz Quiz is a Laravel 12 quiz platform for Admins, Teachers, and Students. It supports quiz publishing, question banks, premium quiz payments, mobile-friendly quiz play, and a reusable JSON API for websites, mobile apps, and partner integrations.

Built For

Admin dashboards, coaching institutes, mobile apps, web frontends, and third-party integrations.

Deployment Friendly

Works on VPS, cloud servers, and shared hosting with a fallback root loader for hosts that cannot point directly to public.

Reusable API

JSON-first endpoints with bearer authentication, pagination metadata, and centralized API error handling.

Access Levels

Public, authenticated user, teacher or admin, and admin-only route groups.

Recommended workflow: finish deployment first, confirm the website loads with CSS, test login, then configure branding, payments, and API integrations.
Before You Start

Server Requirements

  • PHP 8.2 or higher
  • MySQL 8+ or MariaDB equivalent
  • PDO and PDO MySQL extensions enabled
  • OpenSSL, Mbstring, Tokenizer, XML, Ctype, JSON, Fileinfo enabled
  • Apache or Nginx with URL rewriting support
  • Writable storage and bootstrap/cache folders
  • Ability to upload the public/build folder generated by Vite
Important: if the hosted site loads without CSS, the most common causes are missing public/build files, wrong document root, stale Laravel config cache, or incorrect values in the live .env.
Checklist

Deployment Checklist

1. Prepare Build

Run npm install and npm run build before uploading, then confirm public/build/manifest.json exists.

2. Configure Environment

Copy production values into the live .env. Laravel does not auto-read .env.production.

3. Run Migrations

Run database migrations so sessions, cache, jobs, settings, payments, and API token tables exist.

4. Clear Cache

Run php artisan optimize:clear after deployment or environment changes.

Fast Setup

Quick Start Installation

  1. Upload the project files to your server.
  2. Create a MySQL database and database user.
  3. Upload all generated Vite assets inside public/build.
  4. Copy your production environment values into the live .env file.
  5. Point your domain to the Laravel public directory. If your host cannot do that, use the included root index.php and root .htaccess fallback.
  6. Run php artisan migrate --force.
  7. Run php artisan storage:link.
  8. Run php artisan optimize:clear.
  9. Open the website and confirm the homepage loads with CSS before continuing.
  10. Sign in as admin and configure branding, payment settings, and legal pages.
Shared Hosting

Step-by-Step Shared Hosting Deployment Guide

This guide is written for first-time buyers and freshers who want to deploy the script on cPanel, Hostinger, Namecheap, or similar shared hosting.

Best result: set the domain document root to the project public folder. If that option is not available, this project includes a root loader fallback so the app can still run from the project root.
  1. Download the project package on your local machine and extract it.
  2. On your computer, run npm install and npm run build so the public/build folder is ready.
  3. Open your hosting control panel and create a new MySQL database, database user, and password.
  4. Upload the full project to the server using File Manager or FTP.
  5. If your host supports custom document root, point the domain or subdomain to the project public folder.
  6. If your host only serves public_html or project root, keep the included root index.php and root .htaccess files in place so Laravel requests are forwarded into public.
  7. Rename or copy .env.production to the live .env, then update database credentials, app URL, and mail or payment settings.
  8. Make sure APP_URL matches the real domain, for example https://your-domain.com.
  9. Run php artisan migrate --force from terminal access if your host provides SSH.
  10. If your host does not provide SSH, use the built-in installer flow and finish the database and admin setup from the browser.
  11. Run php artisan storage:link. If symlinks are blocked by the host, manually ensure uploaded files are exposed correctly from public/storage.
  12. Run php artisan optimize:clear after every env change, code upload, or migration update.
  13. Visit the website homepage and inspect whether CSS is loading. If the design is broken, verify public/build/manifest.json and all asset files exist on the server.
  14. Log in to the dashboard and upload branding assets, payment keys, and policy pages.

If CSS Is Missing on Shared Hosting

  • Confirm public/build/manifest.json exists on the server.
  • Confirm the files inside public/build/assets were uploaded.
  • Confirm the site is running from public or the root fallback files are present.
  • Confirm the live .env has the correct APP_URL.
  • Run php artisan optimize:clear.

If Images Are Missing

  • Run php artisan storage:link.
  • Check whether public/storage is a valid symlink.
  • If symlinks are blocked, ask the host for symlink support or expose files manually.
  • Verify uploaded logos and avatars exist in storage/app/public.

Security

File Permissions and Writable Paths

  • storage/ must be writable
  • bootstrap/cache/ must be writable
  • public/storage should be a symlink to storage/app/public
  • public/build must stay readable after upload

If uploads are not visible, verify the storage link from admin settings or run php artisan storage:link. If the host blocks symlinks, contact the hosting provider or expose files through their supported method.

Scalability

How to Keep the Project Scalable and Reusable

Database Sessions and Cache

For shared hosting and simple production environments, database-backed sessions and cache are easier to manage consistently than file-based storage.

Queue Readiness

The project is already prepared for database queue jobs. Move to Redis queues on higher-traffic deployments.

Cloud Media

When scaling to multiple servers, use S3 or another cloud disk instead of local uploaded files.

Reusable API

The JSON API is suitable for mobile apps, external dashboards, and partner portals with bearer-token authentication.

JSON API

API Overview

The platform exposes a full JSON API for authentication, quiz browsing, quiz play, result history, payments, and admin management. The API is designed to be secure, scalable, and reusable across web and mobile clients.

Base URL

https://your-domain.com/api

Auth Format

Authorization: Bearer YOUR_TOKEN

Content Type

application/json for most endpoints, multipart for avatar upload.

Rate Limiting

General API limits apply to all routes, with stricter limits on auth endpoints.

Recommended headers: send Accept: application/json on every request and use Content-Type: application/json for JSON body requests.

Authentication

Auth API

POST /api/auth/login

Authenticate a user and receive a bearer token. This is the main login endpoint for web apps, mobile apps, and third-party clients.

Request

{
  "email": "user@example.com",
  "password": "password"
}

Success Response

{
  "message": "Login successful.",
  "token": "plain-text-api-token",
  "token_type": "Bearer",
  "user": {
    "id": 1,
    "name": "Admin",
    "email": "admin@example.com"
  }
}
POST /api/auth/register

Create a student account and receive a bearer token immediately.

GET /api/auth/me

Returns the authenticated user profile.

POST /api/auth/logout

Deletes the current API token and logs the user out.

Errors and Security

404 JSON Errors

Unknown API routes return JSON instead of an HTML error page.

405 Browser Safety

Opening auth endpoints directly in the browser returns usage instructions instead of raw method errors.

429 Rate Limiting

API and auth routes are rate limited centrally to reduce abuse and brute-force login attempts.

Secure Headers

API responses include hardened headers such as X-Frame-Options, X-Content-Type-Options, and cache restrictions for auth-sensitive responses.

Troubleshooting

Frequently Asked Questions

Why does the hosted site load without CSS?

Check that public/build/manifest.json exists on the server, the full public/build/assets directory was uploaded, the domain points to the correct folder, and php artisan optimize:clear has been run after the latest deployment.

Why does login return a 500 error?

Check pending migrations, clear Laravel caches with php artisan optimize:clear, and make sure the running server uses the same database credentials that exist in the live .env.

Why do uploaded avatars or logos not load?

Make sure public/storage is a valid symlink to storage/app/public or that your host exposes uploaded files correctly if symlinks are disabled.

How should mobile apps store tokens?

Store bearer tokens securely in the platform's secure storage, then send them in the Authorization header on protected requests.
